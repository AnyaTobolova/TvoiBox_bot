# Runbook Soft Launch Mini App

## Зачем нужен этот runbook

Этот документ нужен не для разработки, а для аккуратного проведения `soft launch`, когда `mini app` уже технически готов, но ещё не переведён в полноценный production-контур.

## Перед стартом soft launch

- убедиться, что актуальная версия уже выложена в `dev`;
- убедиться, что `@testtvoy_bot` отвечает и открывает mini app;
- убедиться, что `https://app.anyatobolova.ru/mini-api/health` отвечает `200 OK`;
- убедиться, что fallback в основной бот остаётся доступным;
- заранее определить 2-3 клиентов первого круга.

## Кого подключаем первым

- тренер;
- 1 клиент с Android;
- 1 клиент с iPhone, если появится такая возможность;
- 1 резервный клиент, готовый пройти сценарий переноса или отмены.

## Что делаем в первый день

1. Даём участникам soft launch инструкцию входа только через `@testtvoy_bot`.
2. Просим не смешивать тестовые действия в mini app и обычные действия в production-боте в рамках одной и той же новой записи.
3. Проходим минимум такие сценарии:
   - клиент создаёт новую запись;
   - тренер подтверждает запись;
   - клиент открывает `.ics`;
   - тренер отменяет одну тренировку;
   - одна прошедшая или отменённая запись проверяется на уход в архив.
4. Все наблюдения сразу фиксируются в `Этапы разработки.md`.

## Что считается нормальным во время soft launch

- мелкие UX-замечания;
- единичные визуальные правки;
- необходимость переоткрыть mini app после свежего `dev`-deploy;
- отображение reminders `30 минут` в Google Calendar UI тренера при том, что по API уже стоят `1 день` и `1 час`.

## Что считаем стоп-сигналом

- mini app перестал открываться у тренера или клиентов;
- нельзя создать новую запись;
- тренер не видит заявки;
- подтверждение, перенос или отмена ломают состояние записи;
- `.ics` перестал открываться или отдаваться;
- Telegram-уведомления перестали доходить;
- API health перестал быть `200 OK`.

## Что делать при стоп-сигнале

1. Остановить новые действия в mini app для soft launch.
2. Перевести участников обратно в основной бот.
3. Зафиксировать:
   - дата и время;
   - роль (`клиент` / `тренер`);
   - устройство;
   - точный сценарий;
   - скрин и текст ошибки, если есть.
4. Исправление сначала делать в `dev`, потом повторно проверять на ограниченном круге.

## Минимальный шаблон фиксации замечания

```md
- Soft launch observation (YYYY-MM-DD HH:MM MSK)
  - роль: клиент / тренер
  - устройство: Android / iPhone / desktop Telegram
  - сценарий: запись / подтверждение / перенос / отмена / .ics / архив
  - результат: ok / bug
  - детали: ...
```

## Когда можно считать soft launch успешным

- тренер реально пользуется mini app параллельно с ботом;
- первые клиенты проходят ключевые сценарии без ручного сопровождения на каждом шаге;
- остаются только некритичные замечания;
- возврат в основной бот остаётся резервным, но не нужен как постоянная мера.
